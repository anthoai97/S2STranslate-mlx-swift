# Investigate Choppy Real-Time Translation Playback

Status: needs-triage

## Parent

`.scratch/hibiki-ios-mlx/PRD.md`

## Problem

Real-file French-to-English translation now starts playback earlier with buffered pseudo-streaming, but output voice still breaks/chops during the physical-device run. The app also reports CoreAudio overload/channel-map warnings while MLX inference continues running.

The current user expectation is real simultaneous translation: source file streams into Mimi encode -> Hibiki -> Mimi decode -> AVAudio playback, with translated voice beginning after a small delay and then playing smoothly.

## Latest observed run

Configuration:

- Physical-device real-file translation path.
- Source sample: French Europarl short fixture.
- Playback sink: `BufferedStreamingAudioPlaybackSink(prebufferDurationSeconds: 2)`.
- Decoder sample rate: 24 kHz.
- Encoder frame: 1920 samples, approximately 80 ms per source chunk.

## Working hypotheses

- The model path is still sub-realtime on device. A 2s prebuffer drains because generated PCM arrives slower than AVAudio playback consumes it.
- MLX inference and AVAudio scheduling may be contending for CPU/GPU/audio thread time, causing HAL overload and audible gaps.
- The playback engine may be configured in a way that triggers channel-map/converter warnings on device, independent of model speed.
- Event emission/UI updates are reduced and no longer globally main-actor bound, but the run may still do too much synchronous work between cooperative yields.

- [ ] Confirm whether real-time playback remains smooth with a synthetic steady 24 kHz mono PCM stream through the same `AVAudioPlaybackSink`.
